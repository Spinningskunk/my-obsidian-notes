### 一、微信官方机器人：
QQ官方机器人对群主动进行推送、主动私信推送的限制极大，无法满足业务的正常需求，具体限制为：
- 单聊
    - 主动消息每月 `4 条`，超额会发送失败。（例如：给相同用户每月最多发 4 条）
    - 被动消息（回复类）有效时间为 `60 分钟`，每个消息最多回复 `5 次`，超时或超频会发送（回复）失败；
    - 互动召回消息：当用户主动与机器人对话之后每个的周期内可下发 1 条召回消息。周期分别为：当天、1 - 3 天、3 - 7 天、7 - 30 天。在用户隔天发消息给机器人后，周期都会按天维度往后重新计算。
- 群聊
    - 主动消息每月 `4 条`，超额会发送失败。（例如：给相同群每月最多发 4 条）
    - 被动消息（回复类）有效时间为 `5 分钟`，每个消息最多回复 `5 次`，超时或超频会发送（回复）失败；

**最终结论**：几乎无法实现稳定的群聊推送功能，但是因为qq群机器人可以被动呼出、回复消息，如果有类似需求后续可以首选QQ机器人；因为是官方的API所以风险极低；
[QQ机器人官方文档](https://bot.q.qq.com/wiki/develop/api-v2/server-inter/message/send-receive/send.html)

### 二、闲置QQ号+NapCatQQ
用开源项目把代码**伪装成一个正常的、坐在电脑或手机前聊天的真人**。因为普通真人用户发消息是没有“每个月 4 条”这种硬性规定的。
在服务器或电脑上安装一个**正版的官方 QQ 客户端**，然后用开源插件把客户端“劫持”了，暴露出供我们调用的 HTTP/WebSocket 接口。

- **核心原理**：基于腾讯最新一代无 UI 架构的 NTQQ 客户端进行 Hook 注入，对外暴露标准的 OneBot v11 接口。
    
- **选型优势**：
    
    - **不受官方 API 限制**：在腾讯服务器端看来，该机器人的行为等同于一个真实的 PC 端 QQ 用户在进行聊天操作，从根本上绕过了官方机器人的业务额度限制。
        
    - **跨平台与轻量化**：原生支持 Linux (x64/Arm64)，且提供官方 Docker 镜像，支持无头模式（Headless）后台运行，极易集成到现有的服务端架构中。

- **架构设计与部署方案**：
- 推荐采用**“底层协议端 + 业务逻辑端”分离**的微服务架构。

- **底层协议端（通信层）**：
    
    - 在一台 Linux 服务器上通过 Docker 部署 NapCatQQ 容器。
        
    - 配置其开启 HTTP API 服务（例如监听 `3000` 端口）。
        
    - 通过扫码登录一个专门用于推送消息的**闲置个人 QQ 号**。
        
- **业务逻辑端（控制层）**：
    
    - 现有的业务系统不需要集成任何沉重的机器人 SDK。
        
    - 当触发报警或定时任务时，业务系统仅需构建一个极其简单的 JSON payload，通过标准的 HTTP POST 请求发送给 NapCatQQ 容器所在的 `3000` 端口即可。
- **风险评估与应对策略**：
- 虽然脱离了官方 API 的频控，但引入了**腾讯账号安全风控**的新风险。必须采取以下策略保障账号存活率：
	- 1、新号冻结：还是要"养号" ，必须使用有一定注册时长、已实名且有正常好友互动的“老号”。
	- 2、频率过高：如果业务端采用循环无缝发送，瞬间向多个群推送大量消息可能会触发风险，建议**平滑发送**：在业务端的发送逻辑中加入随机延迟（如 `Thread.sleep(1000 ~ 3000)`），避免突发流量
	- 3、内容涉嫌营销：发送大量包含相同外链、敏感词或营销性质的文本；建议跟我们的微信群推送一样，丰富推送文本模板，避免被系统正则匹配拦截。如果是系统通知，尽量只发纯文本或截图，少发外链。
- **最终结论**：**可见，使用NapCat开源项目应该是可以满足推送需求的，且实现简单、风险可控**；
